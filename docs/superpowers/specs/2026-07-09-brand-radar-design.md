## 1. 정체와 경계

### 한 문장

고정된 브랜딩 축을 전제로, 바깥 판(AI/개발 · 이스포츠)에서 지금 무엇이 당겨지는지 조사해,
**"이 주제를 · 이 각도로 · 이 근거로"** 까지 적힌 주제 후보 목록을 내놓는다.

### 축은 상수다 (핵심 벽)

브랜딩 축 = **이스포츠 현직자의 AI 실무 활용**.

- `brand-radar/wiki/axis.md`에 **읽기 전용 상수 한 장**으로 둔다.
- 이 하네스는 축을 **절대 고치지 않는다.** 축이 바뀌면 사람이 `ai-worklog`에서 바꾸고 이쪽에 옮겨 적는다.
- 회의는 "이 축에 무엇이 걸리는가"만 다룬다. **"축이 맞는가"는 안건으로 받지 않는다.**

이 벽이 `ai-worklog`와의 영구 비중첩을 보장한다. 두 하네스는 파이프라인이 다르다
(`ai-worklog`: 내 작업 → 게시물 초안 / `brand-radar`: 외부 자료 → 주제 후보).
유일한 공유 사실인 "포지셔닝 문장"의 소유권을 `ai-worklog`에 둠으로써 SSOT 분열을 막는다.

### 하지 않는 일

- ⛔ 게시물 초안을 쓰지 않는다 (초안은 사람이 쓴다).
- ⛔ 외부에 발행하지 않는다 (외부 발행처 등록 표 전부 빈칸 유지).
- ⛔ 내 작업 기록을 다루지 않는다 (`ai-worklog` 소관).
- ⛔ 축을 개정하지 않는다.

### 상속

SOUL.md · AGENTS.md · CLAUDE.md · 공용 `engine.mjs` · 회고 가드 · 역이식 구조를 킷 그대로 물려받는다.
`engine.mjs`는 바이트 동일 유지. 프로젝트 고유 로직은 `project-walls.mjs`에만 둔다 (CLAUDE §4).

## 2. 한 사이클 (다섯 걸음)

| 걸음 | 커맨드 | 산출 | 상태 전이 |
|---|---|---|---|
| 1 | `/scan` | `signals/<날짜>/` (원본 + `synthesis.md`) | → `scanned` |
| 2 | `/meeting` | 안건 + 기대값 확정 | → `debate` |
| 3 | `/meeting-round` (N회) | `rounds/<날짜>/round-N.md` | `debate` 유지 |
| 4 | `/queue` | `topics/<날짜>-queue.md` | → `report_done` |
| 5 | `/retro` | `retro/<날짜>-retro.md` | → `retro_done` |

각 걸음의 원자 단위는 **산출 + 기록(state/log) + 커밋**이다 (CLAUDE §2-1).

### 불변 계층

`signals/`는 `rounds/`와 동일한 **불변 계층**이다 (`harness.config.json`의 `immutableDirs`에 등록).

이유: 오늘 관측한 소셜 지표는 다음 주면 달라진다. 덮어쓰면 시계열이 사라지고,
시계열이 없으면 **"떠오르는 중"과 "이미 정점"을 구분할 수 없다.** 그 구분이 없으면
하네스는 늘 한 주 늦게 도착한다.

## 3. 수집 계층 — 3중 수집기와 출처 등급

**등급은 사후 라벨이 아니라 수집기가 결정한다.** 어느 수집기가 물어왔느냐로 등급 상한이 정해진다.
(AGENTS.md 62행: 증거 신뢰도 승격 금지 — EXTRACTED / INFERRED / AMBIGUOUS)

### 3-1. `scout-open` — 공개 소스

- 도구: `WebSearch`, `WebFetch`
- 대상: GitHub trending, 블로그 RSS, HN, 공개 릴리스 노트
- **같은 도구, 다른 등급**:
  - GitHub 저장소 별 개수 = 직접 본 숫자 → **EXTRACTED**
  - "요즘 에이전트 하네스가 대세다"라는 블로그 주장 = 남의 주장 → **INFERRED**
- **강제**: 출력 스키마에 `metric` 칸 필수. `metric`이 비면 **등급 상한이 INFERRED로 잠긴다.**
  숫자를 못 봤으면 사실이라 말할 수 없다.

### 3-2. `scout-social` — 로그인 벽

- 도구: `claude-in-chrome` (사용자가 이미 로그인해둔 세션)
- 대상: X · 스레드 · 링크드인 · 인스타그램의 실제 지표(좋아요·리포스트·조회)
- 등급: **EXTRACTED**, 단 조건 둘:
  - `observed_at`(관측 일시) 필수 — 소셜 지표는 시간의 함수다. 날짜 없는 숫자는 숫자가 아니다.
  - 캡처 경로를 남긴다.

### 3-3. `_inbox/` — 사람

- 사용자가 눈에 띈 게시물의 URL·스크린샷을 폴더에 던져둔다.
- 등급: **EXTRACTED** (사람이 직접 관측).
- 에이전트가 아니라 폴더인 이유: 하네스가 못 보는 곳(비공개 커뮤니티, 디스코드, 지인 피드)을
  메우는 유일한 통로.

### 3-4. 막으려는 정확히 하나의 실패

브라우저 수집이 실패했을 때 — 로그인이 풀렸든, 셀렉터가 바뀌었든 — 모델이 조용히 웹검색으로
갈아타서 "링크드인에서 이 주제가 인기"라는 문장을 만들어내는 것. 출처는 그럴듯하고 문장은
매끄럽다. 그리고 완전히 지어낸 것이다.

> ⛔ **수집 채널이 실패하면 그 채널의 결과를 다른 채널 결과로 메우지 마라.**
> 실패한 채널은 `signals/<날짜>/FAILED.md`에 실패 사유와 함께 남기고,
> 종합에서 "해당 플랫폼 데이터 없음"으로 명시한다.

### 3-5. `signal-synthesizer` — 종합

셋을 모아 `signals/<날짜>/synthesis.md`로 종합한다.

여기에 AGENTS.md 63행(소표본 임의 문턱 금지)이 물린다. 게시물 세 개가 잘됐다고
"이 포맷이 뜬다"고 쓰는 순간 소표본 임의 문턱이다.

- 종합의 모든 트렌드 주장은 **"이 표본이 아닌 무엇을 보면 이 판단이 뒤집히는가"** 를 한 줄 단다.
- 못 달면 **잠정 플래그**로만 표기된다.
- **잠정 플래그는 회의에서 근거로 쓸 수 없다.**

### 3-6. 기계 강제 (문서만으로는 표류한다)

`project-walls.mjs`에 벽 하나를 넣는다:

> `/meeting` 진입 시 `signals/<날짜>/synthesis.md`의 모든 항목이
> `tier` · `observed_at` · `source` 세 칸을 갖췄는지 검사하고, 하나라도 비면 회의 시작을 차단한다.

킷의 `step-diff.mjs`가 산출 경계를 지키는 것과 같은 방식이다.

### 신호 항목 스키마

```
{
  "id": "<날짜>-<번호>",
  "claim": "<관측된 사실 한 줄>",
  "source": "<URL 또는 _inbox 경로>",
  "observed_at": "<ISO 일시>",
  "tier": "EXTRACTED | INFERRED | AMBIGUOUS",
  "metric": "<숫자 지표 — 없으면 null, tier 상한 INFERRED>",
  "falsifier": "<이 판단을 뒤집는 관찰 — 없으면 잠정 플래그>"
}
```

## 4. 회의 — 패널 · 악마

킷의 `panelist-pm`(사용자 가치)·`panelist-tech`(실현 가능성)는 물음이 어긋난다.
여기서 답할 질문은 "이 기능을 만들까"가 아니라 **"이 주제를 말할까"** 이므로 패널을 갈아끼운다.

### `panelist-audience` — 독자 쪽

- 이 주제는 누구의 어떤 통증을 건드리는가.
- 그 사람이 이걸 읽고 무엇을 얻어 가는가.
- 지금 이 신호가 뜨는 건 사람들이 **답을 찾고 있어서**인가, 아니면 **그냥 다들 떠들고 있어서**인가.
  후자면 붐비는 방에 한 명 더 들어가는 것뿐이다.

### `panelist-voice` — 축 쪽

- `wiki/axis.md`를 유일한 기준으로 삼는다 (읽기만, 수정 금지).
- 이 주제를 **이스포츠 현직자가 말할 때만 생기는 무언가**가 있는가.
- 없으면 그 주제는 아무나 쓸 수 있는 글이고, 아무나 쓸 수 있는 글은 브랜딩에 기여하지 않는다.

### `redsea-devil` — 악마의 변호인 (특화)

`invest-desk`의 `risk-devil`과 같은 방식의 특화. 매 라운드 **마지막**에 발언하며 세 곳만 찌른다:

1. **레드오션** — 이미 백 명이 쓴 주제 아닌가.
   반드시 `signals/`에서 근거를 끌어와야 한다. **심증만으로는 찌를 수 없다.**
2. **대체 가능성** — 이 각도, 정말 당신만 쓸 수 있나.
3. **근거 부실** — 이 주제를 떠받치는 신호가 잠정 플래그뿐인가.

셋 중 **하나라도 뚫리면 그 후보는 큐에 못 오른다.**

## 5. 산출물 — `topics/<날짜>-queue.md`

후보 하나의 모양:

| 칸 | 내용 |
|---|---|
| **주제** | 술어형 한 줄 ("X 하면 Y 한다") |
| **각도** | 이 축에서만 보이는 지점 한 줄 |
| **근거** | `signals/` 항목 ID + 등급 + 관측일자 (인용) |
| **반론** | 악마가 찌른 것 + 살아남은 리스크 |
| **기대값** | 이 글을 올렸을 때 **어떤 관찰이면 먹혔다고 판정하는가** |

- **근거 칸의 효과**: INFERRED만 붙은 후보는 그 사실이 얼굴에 드러난다.
- **기대값 칸의 효과**: 쓰기 **전에** 적는다. AGENTS §3("결과 보고 기대값 정하지 마라")을 주제 단위로 내린 것.
- **후보는 많아야 다섯.** 열 개짜리 큐는 고르는 비용이 쓰는 비용을 넘어서 아무것도 안 쓰게 만든다.
  (이는 판정 문턱이 아니라 산출 범위 상한이다 — AGENTS 63행 적용 대상 아님.)

## 6. 닫는 고리 — 회고

`/retro`가 사이클을 닫을 때, 지난 큐에서 사용자가 **실제로 쓴 주제**가 무엇이었고
기대값이 맞았는지를 한 줄 확인한다.

- 하네스는 사용자가 뭘 발행했는지 모른다 → 이 칸은 **사람이 채운다.**
- 비어 있어도 사이클은 닫힌다.
- **세 사이클 연속 비면 회고가 그걸 지적한다** — 아무도 안 쓰는 큐를 계속 뽑고 있다는 뜻이므로.

## 7. 명시적으로 뺀 것

- **`/scan` 크론 예약**: 브라우저 수집이 사용자의 로그인 세션을 필요로 하므로 무인 실행이
  반쯤만 성공한다. **반쯤 성공한 수집이 가장 위험하다.** 몇 사이클 손으로 돌려 실패 양상을
  관측한 뒤에 붙인다.
- **`ai-worklog`와의 데이터 연결**: 큐를 넘기지 않는다. 사용자가 읽고 손으로 쓴다.
- **포지셔닝 결정 권한**: §1의 축 상수 벽 참조.
- **리서치 프리미티브의 킷 역이식**: 아직 한 번도 안 돌려본 절차를 공통 자산에 넣지 않는다.
  `brand-radar`로 몇 사이클 돌려 안정화된 뒤 `/retro`의 역이식 경로로 올린다.

## 8. 열린 위험

- **`scout-social`의 취약성**: 셀렉터 변경·로그인 만료로 자주 깨질 것이다. §3-4의 벽이
  "조용한 실패"를 "시끄러운 실패"로 바꾸지만, 깨지는 빈도 자체는 첫 몇 사이클의 관측 대상이다.
- **`_inbox/` 고갈**: 사람이 안 채우면 소셜 신호의 폭이 좁아진다. §6의 3사이클 지적이
  이 신호도 함께 잡도록 회고 체크리스트에 포함한다.
