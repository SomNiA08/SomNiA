# /retro — 회고 · 개정 · 역이식

사이클을 닫는 유일한 문이다. 회고 → **사람 승인(사이클당 1회)** → 프로젝트·원본 킷 동시 반영.

**원본 킷 위치: 형제 디렉토리 중 README.md 첫 줄이 `# harness-kit`인 폴더** (드라이브·폴더명은 컴퓨터마다 달라도 된다 — 이름이 아니라 내용으로 찾는다. 이 프로젝트 자체가 원본 킷이면 자기 자신이 곧 그 폴더다).
**일치가 2개 이상이면 중단하고 사람에게 보고한다** (아래 금지 절 — 첫 일치 채택 금지).

## 절차

1-0. **작동한 벽 실측 (회고의 첫 표 — 복기보다 먼저)**:
   - ⛔ **이번 사이클에 어떤 벽이 실제로 실행됐는지 실측하지 않은 채 회고를 시작하지 마라.** 회고의 첫 표는
     "실패 목록"이 아니라 **"작동한 벽 목록"**이다 — 훅 4종·기계 검사(spec-check·step-diff)·게이트 각각에 대해
     실행 여부와 그 근거를 적는다. 작동하지 않은 벽이 있으면, 그 벽이 담당하던 실패 유형은 **"재발 없음"이 아니라
     "측정 불가"**로 판정한다. 관측기가 꺼진 사이클의 무사고 판정은 판정이 아니라 공백이다.
   - 이 표를 retrospector 브리프 최상단에 넣고, 회고 파일 앞머리에 그대로 기재한다.
   [승격 후보: 절차 1-0에 `checks/hooks-alive.mjs`(CLAUDE §4 v0.40 승격 후보 ②) 호출을 박고, 그 출력표를 회고
   앞머리에 **필수 기재**로 규정. 총괄표의 "재발 판정" 열은 해당 벽이 미실행이면 **"측정 불가"만 허용**하고
   "재발 0"을 쓸 수 없게 한다]
   (v0.42 — 출처: ai-worklog cycle 5, retro/2026-07-12-retro-cycle5.md 메타 관측 · 개정안 7)
1. `state/state.json` 확인 — `status`가 `report_done`이 아니면 안내 후 중단.
   - ⛔ **훅이 로드됐는지 확인하지 않은 채 하네스 커맨드를 실행하지 마라.** 프로젝트 에이전트가 레지스트리
     미등록이면(v0.12 폴백 조건) 그것은 **이 세션의 프로젝트 루트가 이 리포가 아니라는 뜻**이고, 그때
     `.claude/settings.json`의 훅 4종은 **전부 죽어 있다** — 벽2·벽5·slop·사건 장부·회고 가드가 없는 세션이다.
     폴백으로 진행하기로 했으면 착수 전에 (a) 죽은 벽 목록을 명시하고 (b) 훅이 대신하던 기계 검사
     (`spec-check`·`step-diff`)를 **수동으로 반드시** 돌리고 (c) 그 사실을 `log.md`와 회고에 기재한다.
   - ⛔ **빈 사건 장부를 "무사고"로 읽지 마라.** 장부가 비었으면 먼저 **"훅이 돌았는가"**를 확인한다.
     훅이 안 돈 세션의 빈 장부는 무사고의 증거가 아니라 **관측 부재의 증거**다 (SOUL §1 — 부재는 증명한 뒤에만).
     — 이 커맨드에서 이 벽은 1-0의 실측표와 1-1의 장부 집계를 **함께** 읽으라는 뜻이다: heartbeat 없는 빈
     장부는 "사고 0"이 아니라 "관측 0"이다.
   - 폴백 호출 시 `model:` 복원 의무는 CLAUDE §3(v0.41) · `MODELS.md` 적용 방법 4.
   (v0.40 — 출처: ai-worklog cycle 5, retro/2026-07-12-retro-cycle5.md 실패 1)
1-1. **사건 장부 집계 (자동 회고 재료 · HANDOFF W4)**: `state/incidents.jsonl`이 있으면 읽어,
   이번 사이클에 훅이 남긴 사건(pre-tool-use 차단 · post-tool-use 경고 · stop-guard 포기)을
   유형·빈도로 집계한다. 이 집계를 retrospector 에게 넘겨 **실패 후보의 1순위**로 삼는다
   (사람 기억이 아니라 기록이 회고의 시작점이다). 장부가 없거나 비었으면 이 단계는 건너뛴다.
   - **heartbeat 신선도 판독 (v0.47 신설 — 집계보다 먼저)**: 장부의 **마지막 `kind:"session-start"` 줄의 `cycle`**을
     읽어 `state.cycle`과 **대조**한다. 같으면 "이번 사이클에 훅이 돌았다", 다르거나 없으면 **"훅 미작동"**이다 —
     그 판정을 1-0 실측표의 `session-start` 행에 **근거(ts·cycle)와 함께** 적는다.
     판독 조건은 "heartbeat 있음"이 아니라 **"이번 사이클의 heartbeat 있음"**이다 (금지 절).
     `verify:true` 플래그가 붙은 줄은 **수동 검증 실행**이므로 세션 생존의 증거로 **세지 않는다**.
   - **장부의 사건 줄은 이 판정 뒤에만 집계한다.** 훅이 안 돈 사이클의 "사건 0"은 무사고가 아니라 **관측 0**이다.
   (v0.47 — 출처: ai-worklog cycle 7, retro/2026-07-12-retro-cycle7.md 실패 1)
2. `retrospector` 에이전트 실행 → `retro/<날짜>-retro.md` 작성.
   구조: 실패 → 어느 규칙의 구멍 → 재발 방지 금지문 초안 (증거 경로 필수, retrospector 규칙).
3. **개정안 승인 (사이클에서 유일한 사람 개입 지점)**:
   - 개정안이 있으면 사용자에게 항목별로 제시하고 두 가지를 받는다 (AskUserQuestion 활용):
     ① 승인/기각 · ② 적용 범위 = **킷 공통**(모든 프로젝트에 유효한 벽) / **프로젝트 전용**(이 도메인에만 유효).
   - 범위가 애매하면 기본값은 **프로젝트 전용** — 킷 오염보다 킷 누락이 낫다. 다음 프로젝트에서
     같은 구멍이 또 나오면 그때 공통으로 승격한다.
   - 승인된 개정안마다 **한 가지를 더 판정한다**: "이 벽을 **코드·테스트로 승격**할 수 있는가?"
     (훅 패턴 · 게이트 스크립트 · 회귀 테스트). 전용 구현이라도 **추출할 공통 패턴**이 있으면 병기한다.
     산문 벽은 약한 모델이 못 지킬 수 있지만 기계 벽은 지킨다 — 승격 가능하면 개정문에
     `[승격 후보: <방법>]`을 남긴다. (HANDOFF W4 · F2 재발 방지)
   - 개정안이 없으면 이 단계 생략.
4. **승인된 개정안 반영 (반자동 역이식)** — 항목마다:
   - (a) 이 프로젝트의 해당 문서(SOUL/AGENTS/CLAUDE.md)에 Edit로 append — 공통·전용 모두
   - (b) **킷 공통 판정 항목만** 원본 킷(형제 디렉토리 중 README `# harness-kit` 폴더)의 같은 문서에 Edit로 append
     (현재 프로젝트가 원본 킷 자체면 (a)와 동일하므로 생략)
   - (c) 반영한 곳의 `CHANGELOG.md`에 이력 append: 날짜 · 출처 프로젝트 · 범위 · 개정 내용 · 근거(회고 경로)
   - 기각된 개정안은 회고 파일에 "기각 + 사유"로 남긴다 (Edit append).
5. **원본 킷 커밋·푸시 (git 이력 = 자동 백업)** — 역이식이 1건이라도 있으면 즉시:
   5-0. **전파 실측 게이트 (v0.48 신설 — 커밋보다 먼저. "보냈다"가 아니라 "같아졌다"를 센다)**:
   - 역이식 대상 파일마다 **함대 전수 검색 + 해시 대조**를 **커밋 전에 출력**한다:
     그 파일을 보유한 리포 목록 · 사본 수 · 정규화 해시(NormHash). 등록부 정본은 `somnia-hub/bootstrap.ps1`의
     `$repos`다 — 기억·이 파일의 목록을 정본으로 쓰지 마라.
   - **벌 수가 등록부의 보유 리포 수와 다르거나 해시가 갈라지면 그 개정은 미완이다 — 커밋으로 닫지 마라.**
     킷 정본에서 고치고 **나머지에 바이트 복사** → 재대조 → 단일 해시를 확인한 뒤에만 커밋한다.
   - **검사기는 이미 존재한다**: `somnia-hub/audit.ps1` [2](공유 파일 해시 단일 검사)가 정확히 이 검사를 한다 —
     `/retro`는 그것을 **호출**한다. (`audit.ps1` 자체의 개정은 somnia-hub 소유 → 인계 절 6-1.)
   - **적용 대상 판별**: 그 벽이 들어간 파일이 **바이트 동일 공유 파일**(SOUL·TOP-WALLS·MODELS·`engine.mjs`·
     `ledger.mjs` 계열)이면 **전 보유 리포 바이트 동일**이 완료 조건이다. 리포별 특화 파일(CLAUDE·AGENTS·
     커맨드·에이전트 브리프)이면 **파일 전체가 아니라 그 ⛔ 블록의 텍스트가** 바이트 동일이어야 한다 —
     특화 파일에 킷 정본을 통째로 덮어쓰면 그 리포의 고유 벽이 사라진다.
   (v0.48 — 출처: ai-worklog cycle 7, retro/2026-07-12-retro-cycle7.md 실패 2)
   - 원본 킷에서 `git add -A` → `git commit`(메시지: `retro: <프로젝트> cycle N — <개정 한 줄 요약>`)
   - `git push origin main` (일반 푸시만) → 푸시 성공을 출력으로 확인.
   - **1 개정 = 1 커밋** — 개정 전 버전은 git 이력에 전부 남는다. 잘못된 개정의 복구는
     `git revert <커밋>` 한 번이면 된다 (이력이 곧 백업이므로 별도 백업 불필요).
6. 마무리:
   6-0. **승격 이행 검증 게이트 (v0.45 신설 — 이행의 집행자는 여기다)**:
   - 직전 회고 총괄표의 **"승격 이행상태" 열**을 읽는다. 허용 값은 **"이행(+커밋 해시)"** 또는
     **"유예(+사유 + 다음 재발 시 즉시)"** **둘뿐이다** — **"이행 예정"은 금지 값이다**(금지 절 참조).
   - "이행"으로 표기된 승격마다 **파일 실측(존재·grep) + 커밋 해시**를 확인한다. 미이행이면 **"유예"로 강등**하고
     사유를 적는다. 회고는 미이행을 **발견**할 수 있을 뿐 **이행**할 수 없다(retrospector는 `retro/`에만 쓴다) —
     **이행의 집행자는 이 절차 6이다.** (v0.35의 승격 추적 벽은 추적만 시켰고 이행을 강제하지 않았다 —
     그 결과 ai-worklog는 4사이클 동안 승격 후보 마커를 성실히 달고 코드 승격 0건을 냈다.)
   [승격 후보: 이 게이트를 스크립트로 — 직전 회고 총괄표에서 "이행" 표기 행을 파싱해 각 행의 승격 산출물
   (파일 경로·grep 패턴)이 실재하는지 검사하고, 없으면 **`retro_done` 전이 차단**.
   (`hooks-alive.mjs`(§4 v0.40 ②)가 훅 생존을 보듯, 이것은 **승격 생존**을 본다 — 둘은 같은 스크립트에 넣을 수 있다)]
   (v0.45 — 출처: ai-worklog cycle 6, retro/2026-07-12-retro-cycle6.md 실패 6)
   6-1. **인계 항목 확정 (v0.46 신설 — 파이프라인 밖으로 나가는 문)**:
   - 이번 사이클에 발견된 **"타 리포 결함"**을 목록화하고, 각각에 (a) 대상 리포 (b) 증거 경로(이 리포의 기록
     절 번호) (c) 다음 행동 주체를 명시한 **인계 항목**을 회고 파일의 **`## 인계 (타 리포)` 절**에 남긴다.
   - 발견이 없으면 그 절에 **"해당 없음"을 명시**한다 — 절 자체의 생략은 금지다(생략과 무발견은 같은 얼굴이다).
   - **인계 항목이 없는 발견은 회고와 함께 소멸한다.** 배달(대상 리포의 걸음 열기)은 사람·다른 세션의 일이며,
     하네스는 항목을 만들 수 있을 뿐 배달할 수 없다.
   (v0.46 — 출처: ai-worklog cycle 6, retro/2026-07-12-retro-cycle6.md 실패 7 · `AGENTS.md` 파이프라인 절)
   - `state/state.json`: `retro_path` 기록, `status: "retro_done"`
   - `state/.retro-attempts` sentinel 삭제 (Stop 훅 가드 해제)
   - `state/incidents.jsonl`: 1-1의 집계를 회고 파일에 남겼으면 장부를 비운다(빈 파일로 덮어쓰기) — 다음 사이클용.
     **이 걸음은 선택이 아니다 (v0.47)**: 비우지 않으면 **직전 사이클의 heartbeat가 다음 사이클의 생존 증거로
     위조된다** — ai-worklog cycle 6 회고는 17:31:28에 닫혔고 장부 mtime은 **17:07:46에 얼어 있었으며**, 그 낡은
     1줄이 cycle 7 내내 "훅이 돌았다"로 읽힐 뻔했다. **비운 사실을 출력(크기 0)으로 확인하고 보고에 남긴다.**
     (v0.47 — 출처: ai-worklog cycle 7, retro/2026-07-12-retro-cycle7.md 실패 1)
   - `log.md` 한 줄 append: `- <날짜> cycle N 회고 완료 · 개정 <승인수>/<제안수> (공통 <n>·전용 <n>)`
7. 종료 보고: 회고 경로 · 승인/기각·범위 집계 · 역이식 반영 파일 · 킷 커밋 해시만.

## 금지

- ⛔ 사람 승인 없이 규칙 문서를 개정하지 마라. retrospector의 개정안은 항상 *제안*이다 (AGENTS §1).
- ⛔ **프로젝트 전용 개정을 킷에 역이식하지 마라.** 승인 때 받은 범위 분류를 따른다. 도메인 규칙이
  킷에 새는 순간 킷은 범용성을 잃는다.
- ⛔ **킷 공통 판정 항목의 역이식을 건너뛰지 마라.** 건너뛰면 킷이 성장하지 않는다.
- ⛔ **원본 킷을 수정해 놓고 커밋 없이 끝내지 마라.** 개정과 커밋 사이의 틈이 백업 구멍이다.
- ⛔ **원본 킷에 force-push(`--force`) 하지 마라.** 이력이 곧 백업이다 — 되돌리기는 `git revert`로만.
- ⛔ 개정문을 요약·의역해 옮기지 마라. 승인된 문구 그대로, 반영처에 동일하게 append.
- ⛔ 회고 없이 개정만 하지 마라. 개정안의 출처는 항상 회고 파일이어야 한다.
- ⛔ **형제 탐색에서 README 첫 줄 `# harness-kit` 일치가 2개 이상이면 첫 일치를 채택하지 마라.** 중단하고
  일치한 폴더 목록을 사람에게 보고한다 — tie-break 없는 첫 일치 채택은 역이식 오착지의 기본값이다
