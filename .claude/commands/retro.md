# /retro — 회고 · 개정 · 역이식

사이클을 닫는 유일한 문이다. 회고 → **사람 승인(사이클당 1회)** → 프로젝트·원본 킷 동시 반영.

**원본 킷 경로: `C:\work\harness-kit`** (이 하네스가 어느 프로젝트에 복사돼 있든 역이식은 이 경로로).

## 절차

1. `state/state.json` 확인 — `status`가 `report_done`이 아니면 안내 후 중단.
2. `retrospector` 에이전트 실행 → `retro/<날짜>-retro.md` 작성.
   구조: 실패 → 어느 규칙의 구멍 → 재발 방지 금지문 초안 (증거 경로 필수, retrospector 규칙).
3. **개정안 승인 (사이클에서 유일한 사람 개입 지점)**:
   - 개정안이 있으면 사용자에게 항목별로 제시하고 승인/기각을 받는다 (AskUserQuestion 활용).
   - 개정안이 없으면 이 단계 생략.
4. **승인된 개정안 반영 (반자동 역이식)** — 항목마다 세 곳에:
   - (a) 이 프로젝트의 해당 문서(SOUL/AGENTS/CLAUDE.md)에 Edit로 append
   - (b) **원본 킷** `C:\work\harness-kit`의 같은 문서에 Edit로 append
     (현재 프로젝트가 원본 킷 자체면 (a)와 동일하므로 생략)
   - (c) 양쪽 `CHANGELOG.md`에 이력 append: 날짜 · 출처 프로젝트 · 개정 내용 · 근거(회고 경로)
   - 기각된 개정안은 회고 파일에 "기각 + 사유"로 남긴다 (Edit append).
5. 마무리:
   - `state/state.json`: `retro_path` 기록, `status: "retro_done"`
   - `state/.retro-attempts` sentinel 삭제 (Stop 훅 가드 해제)
   - `log.md` 한 줄 append: `- <날짜> cycle N 회고 완료 · 개정 <승인수>/<제안수>`
6. 종료 보고: 회고 경로 · 승인/기각 수 · 역이식 반영 파일 목록만.

## 금지

- ⛔ 사람 승인 없이 규칙 문서를 개정하지 마라. retrospector의 개정안은 항상 *제안*이다 (AGENTS §1).
- ⛔ 프로젝트에만 반영하고 원본 킷 역이식을 건너뛰지 마라. 건너뛰면 킷이 성장하지 않는다.
- ⛔ 개정문을 요약·의역해 옮기지 마라. 승인된 문구 그대로, 양쪽에 동일하게 append.
- ⛔ 회고 없이 개정만 하지 마라. 개정안의 출처는 항상 회고 파일이어야 한다.
